# コントリビューションガイド

Arges プロジェクトへのコントリビューションを歓迎します！このガイドでは、プロジェクトに貢献する方法について説明します。

## 貢献の方法

### 1. Issue の報告

バグを発見した場合や新機能の提案がある場合は、GitHub Issue を作成してください。

#### バグレポート

以下の情報を含めてください：

- **環境**: OS、Python バージョン、依存関係のバージョン
- **再現手順**: 問題を再現するための具体的な手順
- **期待される動作**: 本来どのように動作すべきか
- **実際の動作**: 実際に何が起こったか
- **エラーメッセージ**: 関連するエラーメッセージやスタックトレース

#### 機能リクエスト

以下の情報を含めてください：

- **ユースケース**: なぜこの機能が必要か
- **提案する解決策**: どのように実装すべきか
- **代替案**: 他に考えられる実装方法
- **追加情報**: 関連するリンクや参考資料

### 2. プルリクエスト

#### 事前準備

1. Issue を作成して議論する（大きな変更の場合）
2. フォークを作成
3. 開発環境をセットアップ

```bash
# フォークをクローン
git clone https://github.com/your-username/arges.git
cd arges

# 開発環境をセットアップ
make setup
source venv/bin/activate

# 上流リポジトリを追加
git remote add upstream https://github.com/mahitotsu/arges.git
```

#### 開発フロー

```bash
# 最新の変更を取得
git fetch upstream
git checkout main
git merge upstream/main

# 新しいブランチを作成
git checkout -b feature/your-feature-name

# 開発
# ... コード変更 ...

# テストを実行
make test

# コード品質チェック
make format
make lint

# コミット
git add .
git commit -m "descriptive commit message"

# プッシュ
git push origin feature/your-feature-name
```

#### プルリクエストのガイドライン

1. **ブランチ命名規則**:

   - `feature/`: 新機能
   - `bugfix/`: バグ修正
   - `hotfix/`: 緊急修正
   - `docs/`: ドキュメント更新

2. **コミットメッセージ**:

   - 英語で記述
   - 明確で説明的
   - 現在形で記述（"Add feature" not "Added feature"）

3. **コード品質**:

   - 全てのテストが通ること
   - `make lint` を通すこと
   - 新機能には適切なテストを追加

4. **ドキュメント**:
   - APIの変更時はドキュメントも更新
   - README.md の更新が必要な場合は含める

## 開発ルール

### コーディング規約

- **Python**: PEP 8 に準拠
- **フォーマッター**: Black（設定は pyproject.toml）
- **インポート**: isort で整理
- **型ヒント**: 可能な限り型ヒントを使用
- **Docstring**: Google スタイル

### テストルール

- **カバレッジ**: 新しいコードは 80% 以上のカバレッジを目指す
- **テストファイル**: `tests/test_*.py` の命名規則
- **テスト名**: `test_*` の命名規則
- **Fixtures**: 再利用可能なテストデータは fixtures として作成

### ドキュメントルール

- **マークダウン**: 全ドキュメントは Markdown 形式
- **言語**: ドキュメントは日本語、コメント・コミットメッセージは英語
- **構造**: 論理的で読みやすい構造を維持

## レビュープロセス

### レビュー観点

1. **機能性**: 意図した通りに動作するか
2. **テスト**: 適切なテストが含まれているか
3. **コード品質**: 読みやすく保守しやすいか
4. **パフォーマンス**: 性能に問題はないか
5. **セキュリティ**: セキュリティ上の問題はないか
6. **ドキュメント**: 適切にドキュメント化されているか

### レビュー後の対応

- レビューコメントに対しては建設的に対応
- 必要に応じてコードを修正
- 議論が必要な場合は Issue で継続

## リリースプロセス

### バージョニング

セマンティックバージョニング（SemVer）に従います：

- **MAJOR**: 破壊的変更
- **MINOR**: 後方互換性のある機能追加
- **PATCH**: 後方互換性のあるバグ修正

### リリース手順

1. `CHANGELOG.md` の更新
2. バージョン番号の更新（`pyproject.toml`）
3. タグの作成
4. GitHub Release の作成
5. PyPI への公開（メンテナーのみ）

## コミュニティガイドライン

### 行動規範

- 敬意を持って接する
- 建設的なフィードバックを提供
- 多様性を尊重
- プロフェッショナルな態度を維持

### コミュニケーション

- **Issue**: バグ報告、機能リクエスト、質問
- **Pull Request**: コード変更の提案
- **Discussions**: 一般的な議論（有効にしている場合）

## よくある質問

### Q: 小さな変更でも Issue を作成すべきですか？

A: タイポ修正や明らかなバグ修正などの小さな変更の場合、Issue なしで直接 Pull Request を作成しても構いません。

### Q: テストはどの程度書く必要がありますか？

A: 新機能には対応するテストを含めてください。既存のテストが壊れないことも確認してください。

### Q: ドキュメントはいつ更新すべきですか？

A: APIの変更、新機能の追加、使用方法の変更がある場合は、対応するドキュメントも更新してください。
